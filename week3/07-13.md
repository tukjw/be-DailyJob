### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: HttpResponse를 만드는 방식을 정적 팩토리 메서드에서 Builder 패턴으로 바꿨다
- 그렇게 판단한 이유: HttpResponse객체를 만들때마다 오버로딩된 create 메서드에 매개변수로만 만드는 방식을 구별을 하다보니, 동적 html 기능 이후로 view까지 매개변수로 들어가며 응답객체를 만들때마다 모호함이 늘어나 불편한점이 많아졌다. createWithHeaders 등의 메서드명으로 해결을 해보려고 시도해보았으나 그래도 모호함은 그대로 남아있어 Builder 패턴을 적용하였다.
- 고려했지만 선택하지 않은 대안: create 메서드 하나에 여러가지 파라미터로 오버로딩된 것 대신 createWithHeader, createWithView 등의 메서드로 이름을 바꾸며 해보려했지만 여전히 불편하였다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: HttpResponse 객체를 생성할때 모호성이 많이 사라져 앞으로 응답하는 코드를 작성할때마다 생성을 명확히 할 수 있다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: String으로 구별하던 HttpMethod를 Enum으로 바꾼 후 부터 정상적인 페이지에 Method Not Allowed가 발생하였다.
- 처음 추측한 원인: Enum으로 바꾸다가 method를 분기하는 로직에 문제가 생긴 것 같았다.
- 원인을 확인하기 위해 시도한 방법: Controller 구현체에 method를 분기하는 코드를 찾아보았다.
- 실제 원인: httpRequest.method().equals("POST") 코드가 수정되지않고 그대로 있었다. Enum으로 변경하고난뒤 기존의 equals 코드들이 자동으로 컴파일 오류가 발생할 줄 알았으나 equals는 Object를 매개변수로 하기때문에 컴파일 오류가 없었고 항상 false가 되었기때문에 Method Not Allowed 코드가 발생한 것이다.
- 해결 방법: 메소드를 비교하는 코드를 Enum과 비교하도록 수정하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: Enum으로 바꾸는 리팩토링을 할때는 기존에 사용되던 코드들을 모두 다 살펴봐야겠다고 생각했다. 

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 정적 팩토리 메서드를 Builder 패턴으로 바꾼 것
- 그 접근이 도움이 된 이유: 유지보수에 좋아진것도 있지만 그동안 Lombok으로만 사용하여 내부구현을 모르던 Builder를 직접구현하며 어떤식으로 구현된것인지 알게되고, Builder의 장단점을 알아보는 시간이었어서 좋았다.
- 다음에도 반복하고 싶은 점: 내부 구현을 아는것이 이해도면에서 좋다고 생각하여 앞으로 궁금한 것은 내부구현을 찾아보고싶다

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: Builder패턴을 적용하지않고 createWithHeaders 등의 정적 팩토리 메서드를 이어가려고 했던점
- 그렇게 된 원인: 현재 HttpResponse를 생성하는 코드는 굉장히 많기때문에 Builder 패턴으로 바꾸게되면은 코드의 수정 비용이 많이 컸다.
- 다음에는 다르게 해볼 구체적인 행동: 처음부터 설계를 잘 해야겠다고 느꼈다.
