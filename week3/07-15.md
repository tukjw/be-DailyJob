### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 게시글, 댓글의 프로필사진을 보여주는 기능을 구현할때, 유저 - 프로필사진 경로를 캐싱을 구현하여 사용하였다.
- 그렇게 판단한 이유: 게시글과 댓글을 보여줄때마다 유저와 조인 하는것과, 게시글과 댓글에대한 유저를 한번씩 조회하는 방법을 고민하였는데, 후자 방법을 더 생각해보니 보통 프로필 사진 변경은 많이하지않고, 한번 조회한 프로필사진 경로는 매번 DB 조회할 필요없이 서버에 캐싱해둬서 응답하게되면 효율적이라고 느꼈다.
- 고려했지만 선택하지 않은 대안: 게시글과 댓글을 조회할때마다 매번 조인하는 방법도 있다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 조인이 없어지므로 캐싱만 잘 되어있으면(유저수가 적은경우) 성능 측면에서 조금 더 좋아질 것이라고 생각한다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 클라이언트가 매우 큰 HTTP 헤더나 Content-Length 값을 보내면 서버가 해당 크기만큼 메모리를 계속 사용하거나 한 번에 큰 배열을 생성할 수 있었다. 이로 인해 서버가 느려지거나 메모리 부족으로 종료될 가능성이 있었다.
- 처음 추측한 원인: HTTP 요청을 읽는 과정에서 문제가있을것이라고 생각했다
- 원인을 확인하기 위해 시도한 방법: HTTP 요청을 읽는 HttpFetcher의 동작을 따라가며 헤더와 본문이 메모리에 저장되는 과정을 확인했다. 또한 허용 크기와 정확히 같은 요청, 허용 크기를 1바이트 초과한 요청을 각각 테스트했다.
- 실제 원인: 헤더를 종료 문자가 나올 때까지 제한 없이 저장하고 있었으며, 검증되지 않은 Content-Length 값을 그대로 사용해 본문 배열을 생성하고 있었다.
- 해결 방법: 헤더는 16KB, 본문은 10MB로 최대 크기를 제한했다. Content-Length를 먼저 long으로 파싱한 뒤 음수와 최대 크기 초과 여부를 검사하고, 검증이 끝난 후에만 배열을 생성하도록 수정했다. 경계값과 초과 요청에 대한 테스트도 추가했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 외부에서 받은 크기나 개수 값이 메모리 할당에 직접 사용되는지 먼저 확인하고, 실제 데이터를 읽거나 메모리를 할당하기 전에 최대 크기 제한이 적용되는지 살펴볼 것이다.
