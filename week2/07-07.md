### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: POST 폼 데이터를 GET과 같은 HttpRequest.query 필드 속에 넣은것
- 그렇게 판단한 이유: 폼 데이터는 GET과 동일한 파싱방식을 사용하고, POST 메소드 사용할때 쿼리파라미터를 작성하는경우는 못본 것 같아 GET과 동일한 필드인 query에 넣었다.
- 고려했지만 선택하지 않은 대안: POST 폼 데이터 전용 HttpRequest객체의 필드 추가
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: POST 폼 데이터 전용 필드를 추가하는것도 방법이겟지만, query필드에서 꺼내는게 일관성 측면에서 더 좋을것이라고 생각했고 구분이 필요하면 코드 내에서 GET/POST를 로직으로 구분하는게 더 나을거라고 판단했다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 소켓 HTTP요청 InputStream을 StringBuilder를 통해 String으로 바꾸는 메소드를 작성했었는데, Broken pipe 에러가 발생하여 POST메소드의 본문을 불러오지 못하는 현상이 발생했다. 
- 처음 추측한 원인: BufferedReader readLine이 빈칸이 될떄까지만 변환하다보니 CRLF를 만나면 종료되어 본문을 불러오지 못하였다.
- 원인을 확인하기 위해 시도한 방법: 테스트를 작성해가며 파싱로직을 계속 작성해보았다.
- 실제 원인: Content-Length를 고려하지않아 본문이 제대로 불러와지지 않는것이었다.
- 해결 방법: 어떤식으로 파싱할지 감이 잘 안잡혀 AI에게 자바로 HTTP POST 요청과 본문을 어떻게 파싱할지 질문했고, AI가 응답한 코드는 조금 이해하기 어렵게 작성되어있어 이해하기 쉽게 코드를 수정하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 네트워크 프로그래밍 등을 하다가 Broken pipe 에러를 만나면은 Content-Length를 유심히 살펴봐야겠다고 생각했다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 세션아이디를 만들때 UUID를 이용하였는데 세션 ID를 생성할때마다 UUID.randomUUID() 메소드를 직접 사용하는게 아닌 SessionManager의 static method로 generateRandomSessionId() 메소드를만들었다.
- 그 접근이 도움이 된 이유: 어차피 return UUID.randomUUID() 밖에 없는 메소드지만, 추후에 세션 아이디를 만드는 로직이 변하면 한곳에서만 수정하면되고, 세션아이디를 생성한다는 의미를 명확히했다.
- 다음에도 반복하고 싶은 점: 구현이 조금 더 들어가더라도 의미를 명확히하는 코드를 작성하고싶다.
