# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 스텝5 완료
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 요청 경로에 따른 처리 리팩토링

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 로그인 실패시 응답 본문을 예외 메세지 그대로 보낸 것
- 그렇게 판단한 이유: 401 상태코드만 보내게되면 프론트에서 로그인 실패 이유를 알 수 없었기 때문이다.
- 고려했지만 선택하지 않은 대안: 401 상태코드를 보내고 로그인 실패 이유를 헤더에 작성하기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 코드로 로그인 실패를 확인하고, 응답 본문을 경고메세지로 띄우기때문에 백엔드와 프론트의 결합도를 낮춘 것 같다.

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: PR을 하기전에 커밋에 특정 기능을 추가한 커밋과 그것을 취소를 한 커밋이 남아있어 두 기록만을 지우려고했는데 어려움을 겪었다.
- 처음 추측한 원인: 커밋간의 의존성 문제 때문인가 싶었다.
- 원인을 확인하기 위해 시도한 방법: AI에게 커밋 기록에서 특정 커밋만 삭제하는법을 물어보았다.
- 실제 원인: AI에게 물어보니 특정 기능을 추가한 커밋과 취소하는 커밋 사이에 의존성이 있는 커밋이 남아있기 때문인 것 같았다.
- 해결 방법: 결국 soft reset을해서 커밋을 다시 작성하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 다음부터는 커밋할때 더욱 작업별로 잘 쪼개고 의존성에대해 잘 생각해봐야겠다고 생각했다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 커밋 기록을 수정하기 애매해서 git을 soft reset해버린것
- 그 접근이 도움이 된 이유: 잘못하다가 지금까지 작업한게 다 날아나는것보단 나아보였다
- 다음에도 반복하고 싶은 점: 다음에는 깃에대한 이해도를 더 높여서 자유자재로 사용하고싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 아직도 Router객체(http요청 uri에따른 로직 처리를 직접하는 객체)의 책임이 너무많았고 이 책임을 분리하는 명쾌한 답을 찾지 못했다.
- 그렇게 된 원인: 어떤 방식으로해야 라우팅처리 역할과 로직처리의 깔끔한 분리일까 고민을 많이하다가 아직도 해결을 못한 것 같다.
- 다음에는 다르게 해볼 구체적인 행동: 객체지향이나 디자인패턴 관련 자료 등을 볼때 왜 구조가 그렇게 만들어졌는지 이유를 더 생각해보며 이해도를 높여야겠다고 생각했다.
