> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 스텝3 완료하기

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: HTTP 파싱/응답/쓰기 각각의 책임을 가지는 객체로 분리
- 그렇게 판단한 이유: 현재 내 코드는 유지보수 하기 좋지않고, 미션이 길어질 것으로 예상되기때문에 리팩토링을 했다. 또한 스쿼드세션과 주간피드백때 배운 내용들을 적용해보고싶었다.
- 고려했지만 선택하지 않은 대안: 리팩토링 하지않고 기능 구현하기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 앞으로 코드를 확장할때 드는 비용과 리팩토링하는 비용이 줄어들것으로 예상된다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: BufferedReader 객체의 readLine에서 (line.equals("") || line == null) 인 조건문을 작성하려고했는데 line == null에대한 ide의 경고가 발생했다.
- 처음 추측한 원인: 원인을 못찾았다. readLine은 읽을 것이 없으면 null을 반환하는걸로 알고있는데, 경고메세지를 읽어도 line == null이 항상 true라는 말만 있었고 왜 그런지 몰랐다.
- 원인을 확인하기 위해 시도한 방법: ai한테 물어봤다.
- 실제 원인: line.equals("") 가 이미 null이 아니여야 사용할수잇는 메소드이기때문에 || 뒤에있는 line == null 은 항상 true라서 ide가 경고를 발생시킨거였다.
- 해결 방법: (line == null || line.equals("")) 로 작성하여 null을 먼저 체크하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: null일수도있는 객체를 더 유심히 봐야겠다고 생각했다.

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 어떤식으로 객체를 분리할지 고민을 많이 한 것이 도움이되었다.
- 그 접근이 도움이 된 이유: 나중에 AI를 활용해서 리팩토링을 하더라도 내가 어떤식으로 리팩토링을 할건지 명령을 내릴때 더 효과있는 명령을 내릴 수 있을 것 같았다.
- 다음에도 반복하고 싶은 점: 객체에 어떤 책임만을 갖게할건지 많이 생각해보고싶다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 책임을 어떤식으로 분리할지에 대해 고민한 시간이 너무 오래 걸린 것 같다.
- 그렇게 된 원인: 리팩토링에 대한 숙련도가 높지 않아서 그렇다.
- 다음에는 다르게 해볼 구체적인 행동: 리팩토링 숙련도를 쌓으며 다음부터는 분리할때 상대적으로 더 빠른 속도로 결정할 수 있게 해보려고 한다.
