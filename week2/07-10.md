# 주간 프롬프트 회고

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트

원본 프롬프트: 자바에서 private 메서드에 중요한 로직이 들어있을 때, 이 메서드를 JUnit으로 직접 테스트해야 하는지 아니면 public 메서드를 통해 간접적으로 테스트해야 하는지 알려줘.

선택한 이유: 자바 테스트 코드 작성 중 private 메서드를 어떻게 검증해야 하는지에 대한 구체적인 고민이 드러난 질문이었기 때문이다. 단순히 문법을 묻는 것이 아니라, private 메서드를 직접 테스트하는 것이 좋은지, public 메서드를 통해 검증하는 것이 좋은지처럼 테스트 대상과 테스트 방식에 대한 판단이 필요한 질문이라 의미가 있었다.

---

### 2. 원본 프롬프트 구조 분석

- 문제 상황: 자바에서 private 메서드에 중요한 로직이 들어있는데, 이 메서드를 테스트 코드에서 직접 테스트해야 하는지 고민하는 상황이었다.
- 원하는 결과를 어떻게 요구했는가: private 메서드를 직접 테스트하는 방식과 public 메서드를 통해 간접적으로 테스트하는 방식 중 어떤 방법이 더 적절한지 알려 달라고 요청했다.
- 필요한 맥락이 포함되어 있는가: 자바와 JUnit을 사용한다는 기본 맥락은 포함되어 있었다. 또한 private 메서드에 중요한 로직이 있다는 상황도 드러났다. 하지만 실제 클래스 구조나 private 메서드가 어떤 역할을 하는지는 구체적으로 포함되어 있지 않았다.
- 판단 기준이나 제약 조건이 있는가: 직접 테스트와 간접 테스트 중 어떤 방식이 좋은지 비교해 달라는 기준은 어느 정도 있었다. 하지만 리플렉션 사용 여부, 설계 변경 가능성, 테스트 코드 품질을 우선할지 같은 세부 기준은 명확하지 않았다.
- 출력 방식이 명확한가: 출력 방식은 따로 지정하지 않았다. 개념 설명, 예시 코드, 권장 방식 비교 중 어떤 형태의 답변을 원하는지는 명확하지 않았다.
- 나의 가설, 우려, 의도가 드러나는가: private 메서드를 직접 테스트하는 것이 맞는지, 아니면 public 메서드를 통해 검증하는 것이 더 좋은지 헷갈린다는 의도는 드러났다. 다만 private 메서드를 억지로 테스트하면 설계가 나빠질 수 있다는 우려는 직접적으로 드러나지 않았다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안

```text
자바에서 private 메서드를 테스트해야 할지 고민하고 있습니다.

JUnit으로 테스트 코드를 작성하는 상황이고, private 메서드에 중요한 로직이 들어 있습니다. 이 메서드를 직접 테스트하는 것이 좋은지, 아니면 public 메서드를 통해 간접적으로 테스트하는 것이 좋은지 알고 싶습니다.

다음 내용을 중심으로 설명해 주세요.

1. private 메서드를 직접 테스트하지 않는 것이 일반적으로 권장되는 이유
2. public 메서드를 통해 private 로직을 검증하는 방법
3. private 메서드에 복잡한 로직이 있을 때 클래스를 분리하는 기준
4. 리플렉션을 사용해서 테스트하는 방법과 단점
5. 간단한 Java/JUnit 예시 코드
6. 실무적으로 어떤 방식을 선택하는 것이 좋은지
7. 위 방법들 외에 더 좋은 방법이나 상황에 따라 추천할 만한 대안이 있다면 함께 알려 주세요.
```

### 4. AI가 개선한 프롬프트 분석

개선된 프롬프트는 원본보다 답변받고 싶은 범위와 기준이 더 구체적이다. 원본은 private 메서드를 직접 테스트할지, public 메서드를 통해 간접적으로 테스트할지 묻고 있었지만, 어떤 관점에서 비교해야 하는지까지는 충분히 드러나지 않았다.

개선된 프롬프트에서는 JUnit으로 테스트 코드를 작성하는 상황과 private 메서드에 중요한 로직이 있다는 맥락을 유지하면서, 답변받고 싶은 내용을 항목별로 구체화했다. 그래서 AI가 단순히 테스트 방법만 설명하는 것이 아니라 테스트 설계 관점에서 답변할 수 있다.

또한 private 메서드를 직접 테스트하지 않는 이유, public 메서드를 통한 간접 테스트, 클래스 분리 기준, 리플렉션의 단점, 예시 코드처럼 답변 항목을 구체적으로 나누었다. 이 때문에 답변이 더 체계적이고 실제 개발 상황에 적용하기 쉬워질 것으로 예상된다.

따라서 개선된 프롬프트는 원본보다 목표, 판단 기준, 출력 형식이 더 명확하며, 단순히 “어떻게 테스트하나”를 넘어서 “어떤 방식이 더 좋은 설계인가”까지 판단할 수 있는 답변을 받을 수 있다.

또한 개선된 프롬프트에는 “더 좋은 방법이나 상황에 따라 추천할 만한 대안이 있다면 함께 알려 달라”는 요청이 추가되었다. 이 덕분에 AI가 사용자가 제시한 선택지에만 제한되지 않고, 클래스 분리, 설계 개선, 테스트 범위 조정 같은 더 적절한 해결 방법까지 제안할 수 있다.
